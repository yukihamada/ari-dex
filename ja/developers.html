システム設計

アーキテクチャ概要

ARIは5層のスタック構成です。各レイヤーはモジュール化されており、独立してアップグレード可能です。

Layer 5 — フロントエンド
React SPA、Mobile SDK、React Widget。ユーザーはシンプルなスワップUIを通じてIntentを表現します。チェーン選択は不要です。
↓
Layer 4 — ゲートウェイ
REST + WebSocket API。Intentの検証、レート制限、Solverネットワークへのルーティングを行います。Permit2署名の検証も処理します。
↓
Layer 3 — Solverネットワーク
競合するSolverが接続された全チェーンにわたって最適な実行ルートを発見します。Dutch Auctionにより各バッチの勝者Solverが決定されます。
↓
Layer 2 — マッチングエンジン
暗号化バッチオークション。閾値復号によりIntentが同時に公開されます。均一清算価格によりMEV抽出を防止します。
↓
Layer 1 — 決済
オンチェーンコントラクト（Ethereum、Arbitrum、Base）。Settlement.solがフィルをアトミックに実行。Vault.solがCLMM流動性ポジションをERC-721 NFTとして管理します。

手数料を獲得

Solverになる

SolverはARIの基盤です。ユーザーのIntentに対する最適な実行ルートをリアルタイムで競い合います。

⚔

Solverの役割

Solverは暗号化されたIntentバッチを受信し、AMM、ブリッジ、プライベート流動性にわたる最適なフィルルートを計算し、マッチングエンジンにソリューションを送信します。最良のソリューションがバッチを獲得します。

⚡

Dutch Auctionメカニズム

各バッチはDutch Auctionで実行されます：価格はSolverに有利な水準から開始し、ユーザーの指値に向かって低下していきます。現在の価格水準で有効なフィルを最初に送信したSolverが落札します。これにより速度と価格品質が奨励されます。

💰

報酬モデル

落札したSolverは、決済した各バッチの取引手数料の80%を獲得します。残りの20%はプロトコルのトレジャリーと$ARIステーカーに分配されます。上位Solverは追加のパフォーマンスボーナスを獲得できます。

参加方法

1

100,000 $ARIをステーキング

Solverは最低100K $ARIトークンをSolverRegistryコントラクトにステーキングする必要があります。このステーキングはパフォーマンスボンドとして機能し、悪意のある行為（無効なフィルの送信など）に対してスラッシング可能です。

2

Solverノードを実行

あなたのインフラ上にari-solverをデプロイします。ノードはSolverネットワークに接続し、バッチデータを受信し、ソリューションを送信します。

3

戦略を構築

カスタムルーティングロジックを実装します。オンチェーンAMM、CEX API、プライベートマーケットメーカー、またはクロスチェーンブリッジに接続します。Solver SDKは各ステップのフックを提供します。

bash # Install and run the solver node cargo install ari-solver # Configure with your staking wallet and RPC endpoints ari-solver init \ --private-key $SOLVER_PRIVATE_KEY \ --rpc-ethereum $ETH_RPC_URL \ --rpc-arbitrum $ARB_RPC_URL \ --rpc-base $BASE_RPC_URL # Start solving ari-solver run --strategy custom --log-level info
ツール

SDK & ツール

RustとTypeScriptのファーストクラスサポート。あらゆるアプリケーションにARIスワップを組み込めます。

Rust

ari-core

コア型、Intent検証、暗号プリミティブ（閾値暗号、Poseidonハッシュ）。他のすべてのARIパッケージが使用する基盤クレートです。

toml [dependencies] ari-core = "0.1"
Rust

ari-solver

完全なSolverノード実装。バッチリスナー、ルートオプティマイザー、ソリューションビルダー、オンチェーン決済サブミッターを含みます。トレイトベースの戦略フックにより拡張可能です。

toml [dependencies] ari-solver = "0.1"
Rust

ari-crypto

閾値暗号（BLS12-381）、Poseidonハッシュ、プライベート決済用のゼロ知識証明生成。SolverとマッチングエンジンGの両方で使用されます。

toml [dependencies] ari-crypto = "0.1"
TypeScript

@ari-dex/sdk

ブラウザとNode.js向けのTypeScript SDK。Intentの送信、マーケットデータのストリーミング、Permit2署名の管理。Rustコアから自動生成された型による完全な型安全性。

bash npm install @ari-dex/sdk
TypeScript

@ari-dex/react

ドロップインのReactスワップウィジェット。テーマ対応、レスポンシブ、ウォレット連携（WalletConnect、MetaMask、Coinbase Wallet対応）。3行のコードで埋め込めます。

tsx import { SwapWidget } from '@ari-dex/react' export default () => <SwapWidget theme="dark" />
typescript // Full TypeScript SDK example import { AriClient, Intent } from '@ari-dex/sdk' const client = new AriClient({ apiKey: process.env.ARI_API_KEY }) // Get a quote const quote = await client.getQuote({ sellToken: 'ETH', buyToken: 'USDC', sellAmount: parseEther('1.0'), sourceChain: 'ethereum', }) // Submit an intent const intent = await client.submitIntent({ sellToken: quote.sellToken, buyToken: quote.buyToken, sellAmount: quote.sellAmount, minBuyAmount: quote.expectedOutput, deadline: Math.floor(Date.now() / 1000) + 300, permit2Signature: await signer.signPermit2(quote.permit2Data), }) // Stream status updates client.onIntentUpdate(intent.id, (status) => { console.log(`Status: ${status.state}`) // pending → matched → settling → filled })
オンチェーン

スマートコントラクト

最小限のオンチェーンフットプリント。複雑なロジックはすべてオフチェーンのSolverで実行され、コントラクトはアトミック決済と流動性管理のみを担当します。

Solidity

Settlement.sol

Solverが送信したフィルをアトミックに実行します。ガスレストークン承認のためにPermit2を使用。Solver署名の検証、デッドライン + スリッページ制約の適用、手数料の分配を行います。

主要関数: settle(), batchSettle(), cancelIntent()
Solidity

Vault.sol

集中流動性マーケットメーカー（CLMM）ボールト。LPポジションはERC-721 NFTとして表現され、それぞれがTick範囲と手数料ティアをエンコードします。片側預入と自動複利をサポートします。

主要関数: mint(), burn(), collect(), rebalance()
Solidity

SolverRegistry.sol

Solverのステーキング、スラッシング、レピュテーションスコアを管理します。Solverは最低100K $ARIのステーキングを維持する必要があります。スラッシング条件：無効なフィル、タイムアウト違反、証明可能な共謀。

主要関数: register(), slash(), withdraw()

データモデル

Intent Schema

ユーザーのIntentを表す正規のRust型。これはスタックの全レイヤーを流れるコアデータ構造です。

rust use ethers::types::{Address, U256, Bytes}; use serde::{Serialize, Deserialize}; /// A user's trade intent — the atomic unit of the ARI protocol. /// Intents are encrypted before entering the mempool and only /// revealed via threshold decryption at batch settlement time. #[derive(Debug, Clone, Serialize, Deserialize)] pub struct Intent { /// Unique intent identifier (keccak256 hash of fields) pub id: [u8; 32], /// The user's wallet address (signer) pub user: Address, /// Token being sold pub sell_token: Address, /// Token being bought pub buy_token: Address, /// Exact amount of sell_token to spend (in wei) pub sell_amount: U256, /// Minimum acceptable buy_token received (slippage protection) pub min_buy_amount: U256, /// Source chain ID (e.g., 1 = Ethereum, 42161 = Arbitrum) pub source_chain: u64, /// Destination chain ID (can differ from source for cross-chain) pub dest_chain: u64, /// Unix timestamp — intent expires after this time pub deadline: u64, /// Nonce to prevent replay attacks pub nonce: u64, /// Intent type: Swap, LimitOrder, TWAP, or DCA pub intent_type: IntentType, /// Permit2 signature for gasless token approval pub permit2_signature: Bytes, /// Threshold-encrypted payload (decrypted only at batch time) pub encrypted_payload: Option<Bytes>, } #[derive(Debug, Clone, Serialize, Deserialize)] pub enum IntentType { /// Market swap — fill at best available price Swap, /// Limit order — fill only at specified price or better LimitOrder { limit_price: U256 }, /// Time-weighted average price over N batches TWAP { num_batches: u32, interval_secs: u64 }, /// Dollar-cost averaging over a schedule DCA { frequency_secs: u64, total_epochs: u32 }, } #[derive(Debug, Clone, Serialize, Deserialize)] pub enum IntentStatus { Pending, // Submitted, awaiting batch inclusion Encrypted, // In encrypted mempool Matched, // Solver found a fill route Settling, // On-chain settlement tx submitted Filled, // Successfully settled Expired, // Deadline passed without fill Cancelled, // User cancelled before batch close }
